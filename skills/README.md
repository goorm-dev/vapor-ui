# Vapor UI 세션 워크플로우 가이드

`/using-vapor` 커맨드로 AI와 함께 UI 코드를 작성하고, 디자인 토큰 일관성을 자동 검수받기 위한 가이드입니다.

---

## 한 줄 요약

`/using-vapor`로 UI를 요청하면 코드가 작성되고, 곧바로 토큰 QA가 자동 실행되어 디자인 시스템 위반을 잡아냅니다. 위반이 있으면 수정 여부를 묻고, 사용자가 결정합니다.

---

## 구성

두 개의 스킬이 한 세트로 동작합니다.

| 스킬          | 역할                                                            |
| ------------- | --------------------------------------------------------------- |
| `using-vapor` | 세션 오케스트레이터. 코드 작성과 QA 호출을 조율                 |
| `token-qa`    | 디자인 토큰 검수기. 하드코딩·잘못된 토큰명·잘못된 컨텍스트 탐지 |

`token-qa`는 단독으로도 `/token-qa` 커맨드로 호출 가능합니다(기존 파일 점검 시).

---

## 설치 (최초 1회)

vapor-ui 레포의 `skills/` 디렉토리를 자신의 프로젝트 `.claude/skills/`로 복사합니다.

```bash
cp -r path/to/vapor-ui/skills/using-vapor  .claude/skills/using-vapor
cp -r path/to/vapor-ui/skills/token-qa     .claude/skills/token-qa
```

Vapor UI 패키지 버전이 올라가면 `token-qa/token-catalog.md`를 다시 동기화해야 토큰 카탈로그가 최신 상태를 유지합니다.

## 사용 방법

### 트리거

다음 중 어느 표현이든 세션이 자동으로 활성화됩니다.

- `/using-vapor` 직접 호출
- "Vapor UI로 ~ 만들어줘"
- "@vapor-ui/core 써서 ~"
- "Vapor 컴포넌트로 폼/페이지/화면 ~"

세션이 시작되면 다음 안내가 출력됩니다:

> Vapor UI 세션을 시작합니다. 코드 작성이 끝나면 디자인 토큰 QA가 자동으로 실행됩니다.

### 후속 요청

같은 세션에서 컴포넌트를 추가하거나 수정해도 동일한 워크플로우가 반복됩니다. "방금 만든 폼에 약관 동의 체크박스 추가해줘" 같은 요청도 똑같이 처리됩니다.

## 워크플로우 4단계

```
┌──────────────────────────────────────────────────────────────┐
│  Phase 0  세션 시작 알림                                       │
└──────────────────────────────────────────────────────────────┘
              ↓
┌──────────────────────────────────────────────────────────────┐
│  Phase 1  Vapor UI 컴포넌트로 코드 작성                          │
│           • 사용자가 명시한 값은 그대로 반영                       │
│           • 작성한 파일 경로를 내부적으로 추적                     │
└──────────────────────────────────────────────────────────────┘
              ↓
┌──────────────────────────────────────────────────────────────┐
│  Phase 2  token-qa 자동 호출                                   │
│           • 작성된 파일들에 대해 디자인 토큰 검수                 │
│           • CRITICAL / WARNING / PASS 리포트 생성                │
└──────────────────────────────────────────────────────────────┘
              ↓
┌──────────────────────────────────────────────────────────────┐
│  Phase 3  결과 보고 + 사용자 결정                                │
│           • CRITICAL 있으면 "수정하시겠어요?" 프롬프트            │
│           • CRITICAL 없으면 ✓ 통과로 마무리                      │
└──────────────────────────────────────────────────────────────┘
```

### 핵심 원칙: 사용자 의도 보존

"배경을 #f5f5f5로 해줘"처럼 사용자가 명시적으로 비-토큰 값을 요청하면, Phase 1에서 그 값을 그대로 코드에 반영합니다. 임의로 `$bg-secondary-100` 같은 토큰으로 치환하지 않습니다.

이유: 사용자는 정확한 색을 원했을 수도 있고, "옅은 회색" 정도의 느낌으로 요청했을 수도 있습니다. Phase 2에서 QA가 위반을 보고하고 **여러 토큰 후보를 제시**하면, 사용자가 최종 결정합니다.

예외: "토큰으로 알아서 적당히 변환해줘"처럼 명시적으로 변환을 요청한 경우에만 선제 치환합니다.

## QA 리포트 읽는 법

### 심각도

| 등급         | 의미                                                     | 행동             |
| ------------ | -------------------------------------------------------- | ---------------- |
| **CRITICAL** | 런타임에 디자인이 깨지거나 토큰 시스템이 무력화되는 위반 | 반드시 수정 권장 |
| **WARNING**  | 권장 패턴에서 벗어남. 당장 깨지지는 않지만 일관성 손실   | 선택적 개선      |
| **PASS**     | 통과한 검사 항목. 어떤 부분이 잘 됐는지 가시화           | —                |

### 주요 검사 항목

| 검사                       | 등급     | 잡는 것                                                                            |
| -------------------------- | -------- | ---------------------------------------------------------------------------------- |
| 하드코딩 색상              | CRITICAL | `#hex`, `rgb(...)`, `hsl(...)` 직타 (단, `var()` fallback은 예외)                  |
| 하드코딩 크기              | WARNING  | `'16px'`, `'1rem'` 같은 픽셀 직타 (단, `var()` fallback·`9999px`·`100%` 등은 허용) |
| inline `style=`            | WARNING  | Vapor 컴포넌트에 `$css=` 대신 `style=` 사용                                        |
| 존재하지 않는 토큰명       | CRITICAL | `$bg-secondary-200` 같이 카탈로그에 없는 이름                                      |
| 잘못된 컨텍스트 토큰 사용  | CRITICAL | 배경 토큰을 텍스트 색에 사용 등 토큰의 `usage.avoid` 위반 (다크 모드/접근성 깨짐)  |
| CSS 변수명/fallback 불일치 | CRITICAL | `var(--vapor-size-space-200, 12px)` 같이 변수와 fallback 값이 안 맞음              |
| Primitive 토큰 직접 사용   | WARNING  | `$basic-blue-500` 직접 참조 (semantic 권장)                                        |

### 토큰 카탈로그 근거

token-qa는 추정하지 않고 `token-catalog.md`을 근거로 토큰 후보를 제시합니다. 카탈로그에 없는 이름은 절대 추천하지 않습니다.

semantic JSON에는 토큰의 의도(`intent`), 사용 가능 컨텍스트(`usage.when`), 사용 금지 컨텍스트(`usage.avoid`)가 함께 명시되어 있어서 "토큰명은 맞지만 잘못 쓰인" 경우까지 잡아냅니다.

## 실제 세션 예시

### 예시 1: 깨끗한 흐름

**사용자:** "Vapor UI로 로그인 폼 만들어줘. 이메일·비밀번호 입력, 로그인 버튼."

**AI:**

> Vapor UI 세션을 시작합니다. 코드 작성이 끝나면 디자인 토큰 QA가 자동으로 실행됩니다.

`LoginForm.tsx` 생성:

```tsx
import { Box, Button, TextInput, VStack } from '@vapor-ui/core';

export function LoginForm() {
    return (
        <Box $css={{ padding: '$200' }}>
            <VStack $css={{ gap: '$150' }}>
                <TextInput placeholder="Email" size="md" />
                <TextInput placeholder="Password" type="password" size="md" />
                <Button colorPalette="primary" size="md">
                    Login
                </Button>
            </VStack>
        </Box>
    );
}
```

**토큰 QA 결과 ✓** — CRITICAL 0건 / WARNING 0건. 디자인 토큰이 일관되게 사용되었습니다.

---

### 예시 2: 위반 발견 → 수정 흐름

**사용자:** "Vapor UI로 카드 만들어줘. 배경 #f5f5f5로."

**AI:** `InfoCard.tsx` 생성 — `$css={{ background: '#f5f5f5' }}` 그대로 반영

**토큰 QA 결과 — CRITICAL 1건**

- **[C1]** `InfoCard.tsx:5` — HEX 색상 `#f5f5f5` 직타
- **수정 후보** (의도에 맞게 선택):
    - `$bg-canvas-200` — Card backgrounds, Sidebar backgrounds, Panel backgrounds
    - `$bg-secondary-100` — Inactive tab backgrounds, Neutral badge backgrounds
    - `$bg-canvas-100` — Root page background

**수정하시겠어요?** (예: "`$bg-canvas-200`으로 바꿔줘" / "그대로 둘게요")

**사용자:** "$bg-canvas-200으로 바꿔줘"

**AI:** 파일 수정 후 QA 재실행 → ✓ 통과.

## 자주 묻는 질문

### Q. HEX를 직접 요청했는데 왜 CRITICAL이 떴나요?

A. 일부러 그렇게 설계됐습니다. AI는 사용자 의도를 임의로 바꾸지 않고 그대로 코드에 반영하되, token-qa가 위반을 정직하게 보고하고 후보를 제시합니다. 최종 결정은 사용자에게 맡깁니다. 정확히 그 HEX를 원했다면 "그대로 둘게요"라고 답하면 됩니다.

### Q. `var(--vapor-size-space-200, 16px)`처럼 토큰 fallback에 픽셀값이 있는데 경고가 안 뜹니다.

A. 정상입니다. `var()` 안의 fallback은 CSS 변수가 로드되지 않을 때의 안전망이므로 허용됩니다. 단, 변수명이 카탈로그에 존재해야 하고, fallback 값이 토큰의 실제 값과 일치해야 합니다. 불일치 시 CRITICAL로 잡힙니다.

### Q. token-qa가 추천한 토큰이 실제로 안 들어있는 경우가 있었어요.

A. 그런 경우가 생기지 않도록 토큰 카탈로그(`token-catalog.md`)를 근거로만 추천합니다. 만약 발생한다면 카탈로그가 패키지 버전과 어긋난 상태일 가능성이 큽니다. Vapor UI 버전이 올라간 뒤 카탈로그를 동기화하지 않았는지 확인하세요.

### Q. 단순 컴포넌트 질문에도 토큰 QA가 동작하나요?

A. 아니요. 코드를 작성/수정하지 않은 단순 질문(props 안내, 사용법 설명 등)은 Phase 2가 생략됩니다.

### Q. 파일 쓰기 권한이 없는 환경에서도 동작하나요?

A. 동작합니다. 파일을 쓸 수 없는 경우 코드를 응답 본문에 인라인으로 출력하고, 의도된 파일 경로를 안내합니다. QA도 인라인 코드 텍스트에 대해 그대로 수행됩니다.
