# React Profiler 테스트 가이드

- 문서 버전: v0.1
- 작성일: 2026-03-04
- 대상: Vapor UI 문서 사이트 (`apps/website`)
- 목적: `memo`/`useMemo`/`useCallback` 등 캐싱이 실제 성능 개선에 기여하는지 검증

## 1. 원칙

- 최적화는 체감이 아니라 수치로 판단한다.
- 한 번에 하나의 변수만 바꿔 비교한다.
- 동일한 사용자 시나리오를 반복 측정한다.

## 2. 준비

1. React DevTools(Profiler 포함)가 설치된 브라우저를 사용한다.
2. 측정 전 브라우저 탭, 확장 프로그램, 백그라운드 작업을 최대한 줄인다.
3. 가능하면 프로덕션 빌드 기준으로 확인한다.

```bash
pnpm --filter website build
pnpm --filter website start
```

개발 모드(`next dev`)는 Strict Mode 영향으로 렌더 횟수가 과대 관측될 수 있으므로 참고용으로만 사용한다.

## 3. 측정 시나리오 (아이콘 페이지 기준)

아래 동작을 한 세트로 고정한다.

1. 초기 진입 후 탭 3회 전환 (`basic -> outline -> symbol`)
2. 검색어 `chev` 입력 (4타)
3. 검색어 `arrow`로 변경 (5타)
4. 검색어 초기화 버튼 클릭
5. 다시 탭 2회 전환

최소 3세트 반복 측정 후 평균을 사용한다.

## 4. Profiler에서 확인할 항목

1. Commit duration
- 상호작용 1회당 커밋 시간이 줄었는지 확인

2. Render count
- 관심 컴포넌트가 불필요하게 반복 렌더되는지 확인

3. Why did this render?
- `props changed` / `state changed` 원인을 확인

4. Ranked view
- 시간 소모가 큰 컴포넌트 우선으로 개선 후보 선정

## 5. 캐싱 제거/유지 판정 기준

### `useMemo`
- 유지: 계산량이 크고 의존성 변경 빈도가 낮아 재사용 이득이 명확할 때
- 제거: 계산량이 작거나 의존성 변경이 잦아 재계산 회피 이득이 거의 없을 때

### `useCallback`
- 유지: 콜백을 받는 자식이 `memo` 기반이고 함수 identity 안정화가 실제 렌더 감소로 이어질 때
- 제거: 자식이 `memo`가 아니거나, 유지해도 렌더 횟수/시간 차이가 없을 때

### `memo`
- 유지: 부모 리렌더 상황에서 자식의 불필요 렌더가 확실히 줄어들 때
- 제거: 비교 비용 대비 이득이 없거나 props 변경으로 대부분 다시 렌더될 때

## 6. 실험 절차

1. 기준 브랜치에서 시나리오 3세트를 측정한다.
2. 제거 후보 1개만 수정한다. (예: 특정 `useCallback` 제거)
3. 동일 시나리오 3세트를 다시 측정한다.
4. 평균 수치 비교 후 유지/제거를 결정한다.
5. 다음 후보로 반복한다.

권장: 한 PR에서 최적화와 기능 변경을 섞지 않는다.

| 최대 단일 Commit(ms) |  |  |  | 유지/제거 |
| IconList 렌더 횟수 |  |  |  | 유지/제거 |

추가로 Profiler 스크린샷(변경 전/후)을 PR 본문 또는 코멘트에 첨부한다.

## 8. 의사결정 체크리스트

- 동일 시나리오와 동일 환경에서 측정했는가?
- 평균 기준으로 유의미한 개선이 확인됐는가?
- 코드 복잡도 증가 대비 성능 이득이 충분한가?
- 유지보수성(SRP, 가독성)을 해치지 않는가?

위 항목 중 하나라도 불명확하면 우선 단순한 구조를 선택하고, 추가 데이터 확보 후 재평가한다.
